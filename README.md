## Operation: The Architect's War Room (S16 - OSI Diagnostics & Infrastructure Repair)
**📌 Overview**
This section documents a live-fire infrastructure repair operation. The objective was to enter a sabotaged Linux environment, systematically identify the root cause of multiple system failures using the OSI model, and restore the server to operational status without relying on automated repair tools.

**🛠️ Skills & Methodologies Demonstrated**
* **Layer 7 (Application) - File Permission Auditing:** Audited file metadata using `ls -la` and identified stripped execution bits (`----------`). Restored operability by calculating and applying the correct octal permissions (`chmod 755`).
* **Layer 4 (Transport) - Socket & Process Hunting:** Bypassed standard container troubleshooting when a port collision (`0.0.0.0:8080 address already in use`) was caused by a native host application. Utilized Socket Statistics (`ss -tlnp`) to scan the Linux kernel, identified the rogue Nginx Process IDs (PIDs), and forcefully terminated them (`kill -9`) to free the doorway for production.
* **Layer 3 (Network) - Outbound Traffic Filtering:** Audited the internal host firewall (UFW) to resolve complete external connectivity loss (`ping 8.8.8.8` dropping). Identified and removed `DENY OUT` routing rules.

## Operation: The Hardened Outpost (S18 - Mid-Term Capstone)
**📌 Overview**
Acted as the sole engineer tasked with standing up, securing, and automating a multi-tier containerized infrastructure for a simulated business environment. 

**🛠️ Skills & Methodologies Demonstrated**
* **Perimeter Security:** Hardened the SSH configuration by disabling `PermitRootLogin` and `PasswordAuthentication`. Deployed a `default-deny` UFW firewall policy, explicitly allowing only required ports.
* **Automated Auditing (Python):** Engineered a Python-based watchdog script utilizing the `os` module to monitor disk space (`df -h`) and verify cross-network ICMP connectivity, securely writing output to `/var/log`.
* **Containerized Air-Gapping:** Architected a Docker Compose stack featuring a web frontend (Nginx) and a database backend (MySQL). Ensured the database had zero external routing by placing it on a strictly `internal: true` Docker network, rendering it invisible to perimeter scans.

**⚠️ Real-World Incident Response (Hypervisor Failure):**
During the database deployment phase, the host machine experienced a physical `VERR_DISK_FULL` hypervisor crash while extracting the massive MySQL image, permanently hanging the deployment and corrupting the `ContainerConfig` blueprint. I successfully remediated this block by:
1. Freeing physical storage on the host drive to unfreeze the VM.
2. Identifying the corrupted image metadata within the Docker engine.
3. Executing a manual tear-down (`docker-compose down`) and forced cache purge (`docker rmi -f mysql:latest`).
4. Redeploying the stack from a clean state to achieve a 100% successful build.
